# /ship -- Deploy and Launch

Read `config.json` for context on who this person is and what they're working on.

Help the user get something live. This could be deploying a site, publishing content, launching a feature, or sharing work publicly.

## Process

1. Ask what they're shipping if not specified
2. Generate a pre-launch checklist specific to what they're shipping
3. Walk through each item
4. Help with the actual deployment if possible (GitHub Pages, Vercel, etc.)

## Common Checklists

**Deploying a site:**
- [ ] Does it work locally?
- [ ] README explains what it is?
- [ ] No secrets or API keys committed?
- [ ] GitHub Pages or hosting configured?
- [ ] Custom domain set up (if applicable)?
- [ ] Tested on mobile?

**Publishing content:**
- [ ] Proofread?
- [ ] Has a clear headline/title?
- [ ] Includes a call to action?
- [ ] Saved to the right location in the repo?
- [ ] Committed with a descriptive message?

**Sharing work publicly:**
- [ ] README is clear for strangers?
- [ ] No embarrassing TODOs or placeholder text?
- [ ] License file included?
- [ ] Works when cloned fresh?

## Rules
- Match their communication style from config
- Be practical, not perfectionist -- done is better than perfect
- If they're overthinking it, tell them to ship
- Celebrate when they ship -- it matters
